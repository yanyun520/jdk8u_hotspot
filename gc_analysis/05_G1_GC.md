# G1 收集器 — Garbage-First 垃圾优先收集器

> 源码路径：`src/share/vm/gc_implementation/g1/`

---

## 一、G1 概述

G1（Garbage-First）是 JDK 7u4 引入、JDK 9 成为默认的垃圾收集器。它将堆划分为**大小相等的区域（Region）**，优先回收垃圾最多的区域，从而实现**可预测的停顿时间**。

**通俗比喻**：G1 就像公寓楼的物业管理——不是一次性打扫整栋楼，而是把楼分成多个区域，优先打扫垃圾最多的区域。住户可以告诉物业"我希望每次打扫不超过 200ms"，物业会据此选择打扫哪些区域。

### 1.1 设计目标

- **可预测的停顿时间**：用户指定最大停顿目标，G1 尽量不超过
- **无碎片**：基于 Region 的复制算法，天然无碎片
- **大堆支持**：适合 6GB+ 的堆
- **逐步取代 CMS**：JDK 9 的默认收集器

### 1.2 选用参数

```bash
-XX:+UseG1GC    # 启用 G1
```

## 二、G1 核心概念：Region

**源码**：`src/share/vm/gc_implementation/g1/heapRegion.hpp`

### 2.1 Region 布局

```
G1 堆布局（Region 化）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ E    │ E    │ S    │ O    │ H    │ E    │ O    │ Free │
│ Eden │ Eden │Surv  │ Old  │Humon │ Eden │ Old  │ Free │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  ←→     ←→    ←→     ←→     ←→     ←→     ←→     ←→
  1-32MB 每个 Region（由 -XX:G1HeapRegionSize 控制）

Region 类型：
  E  = Eden      → 年轻代分配
  S  = Survivor  → 存活对象暂存
  O  = Old        → 老年代
  H  = Humongous  → 大对象（> Region/2）
  Free            → 空闲

Region 大小选择：
  堆大小 < 4GB   → Region = 1MB
  堆大小 4-8GB   → Region = 2MB
  堆大小 8-32GB  → Region = 4MB  
  堆大小 > 32GB  → Region = 8MB
  或由 -XX:G1HeapRegionSize 强制指定
```

### 2.2 HeapRegion 类

**源码**：`heapRegion.hpp`

```cpp
class HeapRegion : public ContiguousSpace {
  HeapRegionType _type;           // Region 类型（Eden/Survivor/Old/Humongous/Free）
  HeapWord* _bottom;              // Region 起始地址
  HeapWord* _end;                 // Region 结束地址
  HeapWord* _top;                 // 分配指针
  size_t _grains_bytes;           // Region 大小
  
  // 记忆集（Remembered Set）
  HeapRegionRemSet* _rem_set;     // 跨 Region 引用记录
  
  // 存活率统计
  size_t _live_bytes;             // 存活字节数
  size_t _garbage_bytes;          // 垃圾字节数
  
  // 回收集合
  bool _in_collection_set;        // 是否在回收集合中
};
```

### 2.3 Humongous Objects（大对象）

```
大对象定义：大小 > Region 宸度 / 2

大对象处理：
1. 跨多个连续 Region 分配
2. 直接分配在老年代 Region
3. 回收时作为整体回收

┌──────┬──────┬──────┬──────┐
│ H    │ H    │ H    │ Free │   大对象占3个Region
│(cont)│(cont)│(cont)│      │   连续分配
└──────┴──────┴──────┴──────┘

问题：大对象过多会占用大量 Region，导致年轻代空间不足
解决：尽量控制大对象数量，或增大 Region 大小
```

## 三、G1 收集过程

### 3.1 Young GC（纯年轻代收集）

**源码**：`g1CollectedHeap.cpp` → `do_collection_pause()`

```
Young GC 只回收年轻代 Region：

┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ E*   │ E*   │ S*   │ O    │ O    │ E*   │ O    │ Free │
│ 回收 │ 回收 │ 回收 │ 保留 │ 保留 │ 回收 │ 保留 │      │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  ↑ 标记的 Region 会被回收
  存活对象被复制到新的 Survivor Region 或晋升到 Old Region

步骤：
1. STW 暂停
2. 扫描根集
3. 并行复制存活对象到 Survivor/Old
4. 回收所有 Eden 和 From Survivor Region
5. 更新记忆集
```

### 3.2 并发标记周期（Concurrent Marking Cycle）

**源码**：`concurrentMark.cpp/.hpp`, `concurrentMarkThread.cpp/.hpp`

G1 的并发标记类似 CMS，但基于 Region：

```
┌─────────────────────────────────────────────────────────────────┐
│                     G1 并发标记周期                                │
│                                                                  │
│  ① 初始标记    ─── STW ───  （借道 Young GC）                    │
│       │                                                          │
│  ② 并发标记    ─── 并发 ───  （扫描整个堆）                       │
│       │                                                          │
│  ③ 最终标记    ─── STW ───  （处理 SATB 缓冲区）                 │
│       │                                                          │
│  ④ 筛选回收    ─── STW ───  （回收垃圾最多的 Region）             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 阶段①：初始标记（Initial Mark）

**源码**：`ConcurrentMark::checkpointRootsInitial()`

```
通常随 Young GC 一起执行（借道/piggyback）
标记从根集直接可达的对象
耗时极短
```

#### 阶段②：并发标记（Concurrent Mark）

**源码**：`ConcurrentMark::markFromRoots()`

```
与应用线程并发执行
使用 SATB（Snapshot At The Beginning）保证一致性
遍历整个堆的对象图
使用每个 Region 的 next 位图进行标记

SATB 写屏障：
  每次引用写入前，将旧引用记录到 SATB 队列
  → 保证标记开始时的快照一致性
  
  源码：g1SATBCardTableModRefBS.hpp
  → G1SATBCardTableModRefBS::write_ref_field_pre()
```

#### 阶段③：最终标记（Final Mark / Remark）

**源码**：`ConcurrentMark::checkpointRootsFinal()`

```
STW 暂停
处理 SATB 缓冲区中剩余的引用变更
统计每个 Region 的存活比例
计算 Region 的回收价值（垃圾字节数 / 回收时间）
```

#### 阶段④：筛选回收（Cleanup / Mixed Collection）

**源码**：`G1CollectorPolicy::finalize_cset()`

```
根据停顿时间目标，选择回收价值最高的 Region
混合收集：同时回收部分老年代 Region + 年轻代 Region

 ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
 │ E*   │ S*   │ O*   │ O    │ O*   │ O    │ H    │ Free │
 │ 回收 │ 回收 │ 回收 │ 保留 │ 回收 │ 保留 │ 保留 │      │
 └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
   ↑ 年轻代全回收    ↑ 垃圾多的老年代回收   ↑ 垃圾少的保留

垃圾优先（Garbage-First）的含义：
  优先回收垃圾比例最高（回收价值最大）的 Region
```

### 3.3 Mixed Collection（混合收集）

```
在并发标记周期完成后，G1 可能进行多次 Mixed Collection：

并发标记周期结束
    │
    ▼
Mixed Collection 1: 回收垃圾最多的 N 个 Old Region + 所有 Young Region
    │
Mixed Collection 2: 回收垃圾次多的 M 个 Old Region + 所有 Young Region
    │
    ... (直到回收集合中候选老年代 Region 全部回收)
    │
    ▼
回到纯 Young GC 模式
```

### 3.4 Full GC

**源码**：`g1MarkSweep.cpp` → `invoke_at_safepoint()`

```
G1 的 Full GC 使用标记-压缩算法（类似 Serial Old）
退化为全堆压缩，停顿时间很长
应该尽量避免！

触发条件：
1. 对象分配失败（Eden 和 Survivor 都满，且无法扩展）
2. Humongous 分配失败
3. 并发标记失败
4. 元空间不足
```

## 四、G1 记忆集（Remembered Set）

**源码**：`heapRegionRemSet.cpp/.hpp`, `g1RemSet.cpp/.hpp`

### 4.1 为什么需要记忆集？

```
问题：Young GC 只扫描年轻代，但老年代可能有引用指向年轻代

 ┌─────────────────────────────────────────────┐
 │ Old Region          │  Young Region          │
 │ ┌──────┐           │  ┌──────┐             │
 │ │ Old  │──────────→│  │Young │  跨代引用!  │
 │ │Object│           │  │Object│             │
 │ └──────┘           │  └──────┘             │
 └─────────────────────────────────────────────┘

解决：每个 Region 维护一个记忆集（Remembered Set）
记录"谁引用了我"（其他 Region 中指向本 Region 的引用）
```

### 4.2 记忆集结构

```
HeapRegionRemSet 结构：
┌─────────────────────────────────────────────┐
│  Per-Region Table (PRT)                     │
│  ┌─────────────────────────────────────────┐│
│  │ Region 5 → [card 12, card 45, card 67] ││
│  │ Region 18 → [card 3, card 89]          ││
│  │ Region 42 → [card 7]                   ││
│  └─────────────────────────────────────────┘│
│                                             │
│  Sparse PRT (稀疏表)                        │
│  → 引用少时使用，节省空间                    │
│                                             │
│  Fine-grained PRT (细粒度表)                │
│  → 引用多时使用，卡片级精度                  │
│                                             │
│  Coarse-grained PRT (粗粒度表)              │
│  → 引用极多时使用，Region 级精度              │
└─────────────────────────────────────────────┘
```

### 4.3 写屏障

**源码**：`g1SATBCardTableModRefBS.hpp`

```
G1 使用两种写屏障：

1. SATB 写屏障（并发标记使用）：
   - 在引用被覆盖前，将旧引用入队到 SATB 队列
   - 保证标记开始时的快照一致性
   - 源码：satbQueue.cpp

2. Refine 写屏障（跨 Region 引用更新）：
   - 在引用写入后，更新目标 Region 的记忆集
   - 使用 Dirty Card Queue
   - 源码：dirtyCardQueue.cpp, concurrentG1Refine.cpp
   - 后台线程 ConcurrentG1RefineThread 持续处理脏卡
```

## 五、G1 回收集合选择（Collection Set）

**源码**：`g1CollectorPolicy.cpp/.hpp`, `collectionSetChooser.cpp/.hpp`

### 5.1 收集策略

```
G1CollectorPolicy 根据以下因素选择回收集合：

1. 停顿时间目标（MaxGCPauseMillis）
   → 每次收集不能超过目标时间

2. 回收效率（垃圾字节数 / 预估回收时间）
   → 优先选择效率最高的 Region

3. 混合收集候选集
   → 并发标记后，存活率低的 Old Region 进入候选集
   → 每次混合收集选择部分候选 Region

选择算法：
  1. 年轻代 Region 全部进入回收集合
  2. 按 GC 效率排序候选老年代 Region
  3. 在停顿时间预算内，选择尽可能多的候选 Region
  4. 如果停顿时间预算不够，只回收年轻代
```

### 5.2 MMU 追踪器

**源码**：`g1MMUTracker.cpp/.hpp`

```
MMU（Minimum Mutator Utilization）追踪器：
- 保证应用线程（Mutator）的最小使用率
- 例如：MaxGCPauseMillis=200ms, GCPauseIntervalMillis=1000ms
  → 每秒最多 200ms 用于 GC
  → MMU = 80%（应用线程至少获得 80% 的时间）

g1MMUTracker.hpp:
  class MMUTracker {
    double _max_gc_time;    // 最大 GC 时间
    double _target_pause_time; // 目标停顿时间
    // ...
  };
```

## 六、G1 核心参数调优

### 6.1 基础参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:+UseG1GC` | false（JDK8）/ true（JDK9+） | 启用 G1 |
| `-XX:G1HeapRegionSize` | 自动 | Region 大小（1/2/4/8/16/32MB） |
| `-XX:MaxGCPauseMillis` | 200 | 最大 GC 停顿时间目标(ms) |
| `-XX:GCPauseIntervalMillis` | 0 | GC 停顿间隔目标(ms) |
| `-XX:ParallelGCThreads` | CPU核数 | STW 期间并行线程数 |
| `-XX:ConcGCThreads` | ParallelGCThreads/4 | 并发标记线程数 |

### 6.2 Region 大小选择

```
Region 大小影响：
- 太小：Region 数量多，记忆集开销大
- 太大：Humongous 对象多，浪费空间
- 推荐：Region 数量在 2048-4096 之间

自动选择（默认）：
  堆 2GB  → Region 1MB（2048个）
  堆 4GB  → Region 2MB（2048个）
  堆 8GB  → Region 4MB（2048个）
  堆 16GB → Region 8MB（2048个）
  堆 32GB → Region 16MB（2048个）

手动指定：
  -XX:G1HeapRegionSize=8m
```

### 6.3 年轻代大小

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:G1NewSizePercent` | 5 | 年轻代最小占比(%) |
| `-XX:G1MaxNewSizePercent` | 60 | 年轻代最大占比(%) |
| `-XX:G1MixedGCLiveThresholdPercent` | 85 | 混合收集时 Old Region 存活率阈值 |
| `-XX:G1ReservePercent` | 10 | 保留空间占比(%) |

**重要**：不要设置 `-Xmn`！G1 需要动态调整年轻代大小来满足停顿时间目标。

### 6.4 混合收集

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:InitiatingHeapOccupancyPercent` | 45 | 触发并发标记的堆占用率(%) |
| `-XX:G1MixedGCCountTarget` | 8 | 混合收集次数目标 |
| `-XX:G1OldCSetRegionThresholdPercent` | 10 | 每次混合收集最大 Old Region 占比 |
| `-XX:G1HeapWastePercent` | 5 | 放弃混合收集的浪费率阈值 |

### 6.5 大对象

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:G1HeapRegionSize` | 自动 | Region 大小（影响大对象阈值） |
| `-XX:G1HumongousThresholdPercent` | 50 | 大对象占 Region 百分比阈值 |

### 6.6 完整 G1 调优示例

```bash
# 典型 G1 配置（8核 16GB 堆，延迟敏感应用）
java -XX:+UseG1GC \
     -Xms16g -Xmx16g \
     -XX:MaxGCPauseMillis=200 \
     -XX:ParallelGCThreads=8 \
     -XX:ConcGCThreads=2 \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:G1HeapRegionSize=8m \
     -XX:G1MixedGCLiveThresholdPercent=85 \
     -XX:G1MixedGCCountTarget=8 \
     -XX:+UseStringDeduplication \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar app.jar

# 说明：
# - MaxGCPauseMillis=200 → 目标每次 GC 不超过 200ms
# - InitiatingHeapOccupancyPercent=45 → 堆使用45%时开始并发标记
# - G1HeapRegionSize=8m → 16GB堆约2048个Region
# - UseStringDeduplication → 字符串去重（节省内存）
# - 不设 -Xmn，让G1自动调整年轻代
```

## 七、G1 字符串去重

**源码**：`g1StringDedup.cpp/.hpp`

```
G1 特有功能：字符串去重（String Deduplication）
-XX:+UseStringDeduplication 启用

原理：
  String 对象由 char[] 和 hash 组成
  不同 String 可能内容相同但 char[] 不同
  去重：让相同内容的 String 共享同一个 char[]

 ┌────────────┐     ┌──────────────┐
 │ String "hello" │────→│ char[] "hello" │
 └────────────┘     └──────────────┘
                          ↑
 ┌────────────┐     ┌──────────────┐
 │ String "hello" │────→│ (共享同一个)    │
 └────────────┘     └──────────────┘

源码组件：
- G1StringDedupThread: 后台去重线程
- G1StringDedupQueue: 待去重字符串队列
- G1StringDedupTable: 去重哈希表
- G1StringDedupStat: 去重统计
```
